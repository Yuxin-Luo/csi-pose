1一句话目标

把 6 块 ESP32-S3 路由器摆成 3×3 网格，从 WiFi 信号的微小变化中， 实时"画"出房间里那个人的 18 个关节 2D 坐标—— 不戴设备、不开摄像头、只用路由器。

这是开源项目 csi-pose（论文 Can WiFi Estimate Person Pose?，arXiv:1904.00277）的复现工作。 原项目用的是 Intel 5300 单块网卡，本工作把它移植到 6 块 ESP32-S3 上。 原项目只在 1 个会话（fall-demo-01）跑过跌倒检测，本工作在自己录的 580 秒会话上重新跑通整个 pipeline。

💡 摄像头只在采集阶段用一次——当教师给 CSI 视频"打标签"。 训练好模型后，运行时只要 6 块路由器就够了，不用摄像头。

2项目是什么

WiFi 信号在房间里传播时，会被人体的不同部位反射、散射、吸收。 ESP32-S3 的 WiFi 芯片能输出一种叫 CSI（Channel State Information） 的数据—— 本质上是56 个频段上信号强度的时间序列。 把这串数字扔给一个轻量神经网络，神经网络就能反推出人体的 18 个关节点坐标。

2.1 系统由 5 个子系统组成

① 固件

6 块 ESP32-S3

3 块发射器（TX）广播 WiFi 信标，3 块接收器（RX）抓 CSI 数据流

② 采集

录制与同步

电脑串口收到 RX 的 130B 帧，按时间戳写进 HDF5；同时摄像头录 mp4

③ 教师

摄像头打标签

RTMDet 检测人，RTMPose 估 17 关节，转成 BODY-18 拓扑

④ 训练

CSI → 关节

轻量卷积神经网络，从 50ms CSI 窗口回归出 18 关节 2D 坐标

⑤ 实时

推理 + 跌倒

回放或实时 CSI 流 → 火柴人 → 规则式状态机判跌倒

3用了什么技术（简明）

本节用不深入技术细节的方式讲清项目依赖的几件事。 想看完整技术栈的同事可以参考 技术栈总览文档。

3.1 硬件：6 块 ESP32-S3 摆成 3×3 网格

3 块做发射器（TX），3 块做接收器（RX），物理上分开放置。 每对 (TX, RX) 形成一条无线链路，3×3 共 9 条独立链路—— 相当于一个9 通道的信道矩阵，每条通道都给出一串 CSI 数值。

3.2 时钟同步（最简版）

6 块板子的晶振各自独立、和电脑时钟也不一致。 本项目不做显式的板间同步协议，而是靠两层兜底：

  • TX beacon 同时性：3 块 TX 同时广播，RX 收到时把它们视为"同一物理时刻"
  • 每板下包络拟合：USB 串口延迟只能让包迟到、不能早到，所以散点的下凸包 = 真实时钟线

这套办法精度约 ±15 ms——跌倒瞬间人体速度 2-3 m/s，对应 3-5 cm 标签噪声，足够训练。

5Bug ① · 固件层（4 层 bug 链）

第一块 RX 板插上 USB 后完全抓不到 CSI 帧，看似一个 bug 实际是 4 层 bug 串联：

固件

Bug · 4 层 bug 链：从 CONFIG 到 SDK 编译再到 USB 路由

现象RX 板插上后 rawlog 0 个 130B 帧
根因 1CONFIG_ESP_WIFI_CSI_ENABLED=y 没开 → CSI 接收被关闭
根因 2idf.py build 不会重读 sdkconfig.defaults，需要清理 rebuild
根因 3console 走 UART0 不走 USB，bridge 在 /dev/ttyACM0 听不到
根因 4csi_link 默认 SERIAL_UART0，130B 帧走 GPIO 不走 USB
解决同时改 sdkconfig + sdkconfig.defaults + CONFIG_ESP_CONSOLE_USB_SERIAL_JTAG=y + CONFIG_CSI_LINK_SERIAL_USB_JTAG=y
出处dev_doc/4-rx-firmware-bringup-debug-2026-07-10.md §3.2-§3.4
💡 教训：任何一层只修半截都不行——表面"没数据"，实际是 4 个独立 bug 叠加。 这种"看上去一个 bug 实际一串"的链路在固件里特别常见，必须从物理层一路查到编译配置。

6Bug ② · 物理层（95% loss 误判）

第一次跑 60s 烟雾测试，bridge 报告 95% 丢包率——表面像系统崩溃。

物理

Bug · 95% loss 误判：live.log 累计 + USB 物理抖动

现象live.log 报告 loss=95%，3 块 RX 同一时刻集体断联 18.7 min
真相 1LinkTracker 启动时 first_seq 跳变，把历史 gap 全计入 lost（startup accounting 伪影）
真相 2dmesg 显示 14 秒内 3 个 USB root port reset → Linux xhci + USB autosuspend 物理抖动
解决 1改用 h5 recorder_status.links.*.loss（链路 0-0 = 1.62%）
解决 2disable autosuspend：echo on > /sys/bus/usb/devices/.../power/control（边际改善 ~0.1pp）
出处dev_doc/11-loss-gap-investigation / dev_doc/13-loss-final-verdict
💡 教训：loss 数字本身会骗人——bridge 报的累计数字和实际链路丢包率差出 50 倍。 任何对数字异常的第一反应都应该是"测两个独立链路互相验证"，而不是急着改代码。

7Bug ③ · 训练 / 数据层

训练阶段暴露了 4 个独立的"看着像 bug"的坑：

训练

Bug · rawlog chunked streaming 误读

现象解析器把 1 record 当成 1 帧，结果数据少 80%
根因实测是 chunked streaming，1 chunk 内多帧（5 帧/650B 主流）
解决解析先按 <u64,u16> 拆 chunks，再 payload.find(b"\x1d\xc5") 切帧
出处dev_doc/5-loss-throughput-baseline §3.3
训练

Bug · 训练 OOM @ batch=64 + in_ch=560

现象WiSPPN.forward 上采样到 144×144，中间张量爆显存
根因(B,150,144,144) f32 单 batch ≈ 11.9 MB；batch=512 在 16GB GPU 实测 OOM
解决configs/train-norm.yaml 改 batch=32；未来优化 gradient checkpointing / AMP fp16
出处dev_doc/21-norm-training §4.2
训练

Bug · onnxruntime CPU-only 教师慢 80×

现象RTMDet+RTMPose 教师 41 秒/帧（GPU 没启用）
根因pip 默认 onnxruntime 无 CUDAExecutionProvider
解决pip install onnxruntime-gpu==1.19.2 + nvidia-* 系列 + LD_LIBRARY_PATH
效果GPU 启用后教师 41.9 fps（vs CPU 0.5 fps）
出处dev_doc/21-norm-training §4.1
训练

Bug · 摄像头 mp4 metadata fps=30 但实测只有 15fps

现象摄像头 mp4 报头写 30fps，实际播放只有 15fps
根因USB2 带宽上限；CAP_PROP_FPS 报 30 是协商值不是实测
解决probe 实测 4cc × resolution 矩阵 → 切 MJPG 640×360（实测 28.30fps）
出处dev_doc/12-cam-fps-probe §2

8Bug ④ · 推理层（骨架坍缩对角线）

norm 训练结束得到 PCK@0.5 = 78.9% 的"高精度"模型——但放进实时 demo 里骨架直接塌成对角线。

推理

Bug · 骨架坍缩成对角线（in_ch 不匹配）

现象RT demo 视频里 18 关节全部坍缩到画布角落对角线
根因best.pt in_ch=560（amp+phase+rssi 三路联合），但 RT pipeline ReplaySource 只读 amp (in_ch=280)；placeholder 填零后 z-score 不再零均值
后果xy 坍缩到 ±5,±13 → project clip 到画布角落
解决绕过根因：改用 amp-only 模型 s01-rX-norm-amp/best.pt (in_ch=280)；骨架正常渲染，infer p50 ≈ 10ms
残留amp-only 模型 PCK@0.5 = 55.5%（vs norm 78.9%，差 23pp）
出处dev_doc/22-diag-skeleton-handoff / dev_doc/23-amp-demo-working
推理

Bug · boot 在 RTMDet load 时 timeout

现象远程录制场景下 boot 启动被 RTMDet 模型下载阻塞（~145MB）
根因RTMDet 启动需要下载模型权重
解决commit da7fc65 加 --no-skeleton --no-overlay，让 cam 走最简路径
出处git commit da7fc65

9Bug ⑤ · 流程编排（boot 6 次失败史）

第一次想用一个 bash 脚本把"5 个进程" + "摄像头" + "录制"统一起来，结果脚本 连续 6 个版本全部 silent death。

流程

Bug · boot 主进程 silent death（v1-v6 失败史）

现象启动几分钟后主进程消失、log 卡住、没有任何错误信息
v1-v3polling 循环 silent 死亡（疑似 set -e + && 触发）
v4权限 / trap 处理不当
v5-v6cam/recorder input() 在 & 后台 stdin 非 TTY 时 raise EOFError，set -e 触发退出
真根因Python input() 在后台非 TTY 环境触发 EOFError，bash set -e 默默退出
解决commit 91d09d3：input() → cv2.waitKey() + sentinel flag
经验5 报错就停，质疑方法本身而不是补丁
出处dev_doc/9-handoff-boot-recording-debug
💡 教训：silent death 是最难调的 bug 类型。 看似"没报错"，实际是 shell + Python 的多种微妙交互在后台触发； 必须先写最小复现脚本隔离问题，而不是在原脚本上加更多 set -e 防御。

10方法论：5 个原则

7 天复现下来沉淀的 5 条工作原则（已写入本目录 CLAUDE.md）：

①
5 报错就停
同一方法连续 5 个报错 = 重新审视方法本身，而不是继续打补丁
②
根因优先
先问"为什么"，再问"怎么办"；绕开根因 = 留下未来雷
③
决策可追溯
每个阈值/参数都在 dev_doc 留依据，方便后来人 audit
④
不绕开同步
下包络是核心，不能简化；USB 延迟只能让包迟到不能早到
⑤
诚实声明
单会话标定的阈值、单模型能力的局限要明说，不能夸大
⏸️ 留底：所有 bug、决策、参数都对应一份 dev_doc（编号 1-24），可逐条审计。 工作量事实：23 份过程文档 + 5 个核心 commit + 4 套训练产物。

13诚实声明 · 下一步

13.1 已知局限

⚠️ 必须承认的事实，避免被数字误导：
局限现状影响
amp-only 模型能力差距 PCK@0.5 = 55.5%（vs norm 78.9% 差 23pp） 实时 demo 骨架能用，但精度不如联合训练模型
phase/rssi 实时通路未实现 ReplaySource/LiveSource 只流式读 amp 高精度模型无法直接用于实时推理
单会话标定 所有阈值/教师标签都来自 s01-rX norm 580s 未做多会话、跨用户验证
摄像头 fps 物理上限 USB2 cam 实测最高 15-30fps 想更高需换 USB3 cam
Ctrl-C 强杀 mp4 moov 缺失 ffprobe 报错需手动修 录制脚本需优雅退出
跌倒 FSM 未在多人 / 多场景验证 继承上游 fall-demo-01 单会话标定 不是医疗/安全级设备

13.2 下一步

  1. 实现 phase/rssi 实时通路——接 ReplaySource 按 packet t_ns 查 /samples/X_phase/rssi，让高精度模型能上实时
  2. 多会话 / 跨用户验证——至少 3 个不同人在不同房间重做录制，验证阈值鲁棒性
  3. amp-only 模型继续训练——当前只跑了 13 epoch（被 cut），跑满 30 epoch 看是否能逼近 78.9%
  4. TX2 单板 7.59% loss 根因——晶振个体差异 / esp_timer 漂移，需硬件级排查
  5. 跌倒 FSM 跨用户标定——继承 CLAUDE.md §1.1 诚实声明
